# ADR 010: Logical Focus Session Refactor

## 1. Product Philosophy
Flow Extension is **NOT** a separate focus session. It exists to let users continue working without being interrupted by a rigid timer. Therefore, a Logical Focus Session begins when the user starts a Work phase, and Flow Extension is simply an optional continuation of that identical session.

The application treats a sequence of:
`25m Work + 18m Flow`
as exactly **one 43-minute focus session**.

This matches user expectations: when reviewing their day, they do not think "I did 25 minutes of work, and then 18 minutes of flow." They think "I focused for 43 minutes straight on this task."

## 2. Domain Model
The new Logical Focus Session model tracks two mathematically independent concepts:
1. **Total Focus Duration:** The raw sum of time spent in the session (Work + Flow).
2. **Core Work Completed:** A boolean indicating whether the original target (e.g. 25 minutes) was successfully achieved.

These are intentionally decoupled because stopping a Flow Extension early shouldn't revoke the achievement of completing the core 25-minute goal.

**Example 1:**
`25m Work (Natural) -> 18m Flow (Reset)`
- **Duration:** 43 minutes
- **Core Work Completed:** true
- **Completed Sessions Count:** 1

**Example 2:**
`12m Work (Reset)`
- **Duration:** 12 minutes
- **Core Work Completed:** false
- **Completed Sessions Count:** 0

Duration-based statistics still include the 12 minutes from Example 2 because the user did genuinely focus for 12 minutes, even if they aborted before reaching their goal.

## 3. Session Recording
The atomic building block of history is the `SessionRecord`.
- `continuationOf`: A UUID pointer chaining fragments together if they belong to the same Logical Focus Session (e.g., pause gaps, Flow Extensions).
- `termination`: An explicit enum (`.natural`, `.split`, `.reset`) recording exactly *how* this specific fragment ended.
- `isCoreWorkCompleted`: The canonical centralized business rule. Evaluates to `true` if and only if `phase == .work` and `termination == .natural`.

Continuation chains are built by passing the previous fragment's UUID into `continuationOf`. This pointer is strictly preserved during a `Work -> Flow` transition to fuse them. It is intentionally cleared for all other transitions (Break, Reset, Stop) because those demarcate the end of the logical session.

## 4. ContinuousSession
`ContinuousSession` represents a **Logical Focus Session**, generated by merging one or more `SessionRecord` fragments.
- `duration`: The summed duration of all `constituentRecords`.
- `coreWorkCompleted`: Evaluates to true if *any* constituent fragment achieved `isCoreWorkCompleted == true`.
- `constituentRecords`: The raw array of fragments making up this block.

The legacy `isCompleted` property was removed because it conflated "did the timer finish naturally" with "is this the end of the timeline chunk", which caused severe bugs when Flow Extensions were aborted.

## 5. Statistics
Statistics calculation has been completely stratified:
- **Completed Sessions:** Strictly counts `ContinuousSession`s where `coreWorkCompleted == true`.
- **Total Focus / Longest Session / Averages:** Operate purely on the merged `duration` of the `ContinuousSession` blocks.

This guarantees that a session can contribute 43 minutes to "Total Focus" while cleanly contributing exactly 1 to "Completed Sessions".

## 6. Continuation Chain Lifecycle
The `TimerManager.lastSessionFragmentID` variable holds the active continuation pointer.
It is preserved **only** during the `Work -> Flow Extension` transition.
It is explicitly cleared for transitions into breaks, or when skipping/resetting, because those actions represent the user stepping out of the continuous focus flow, thereby terminating the Logical Focus Session.

## 7. Builder Assumptions
**Architectural Invariant:** `ContinuousSessionBuilder` strictly assumes that `SessionRecord`s are supplied in chronological creation order.
This is currently guaranteed by the `TimerManager` / `HistoryManager` pipeline, which writes and reads records linearly.
*Warning:* If FlowTimer ever introduces cloud sync, import/export, or unordered history ingestion, this builder will silently corrupt session chains. It must be refactored into an order-independent dictionary/graph-based implementation.

## 8. Verification Summary
The implementation was mathematically verified against all known fragmentation scenarios:
- **Normal Work -> Flow -> Break:** Merged cleanly. `coreWorkCompleted = true`.
- **Work -> Flow -> Reset:** Merged cleanly. Aborting flow did NOT revoke the core work completion. `coreWorkCompleted = true`.
- **Reset during Work:** `coreWorkCompleted = false`.
- **Pause-gap split:** Absorbed safely into the session. `coreWorkCompleted = true` if ultimately finished.
- **Sleep/Wake recovery:** Acts identical to pause-gap. Absorbed safely.
- **Crash recovery:** Offline time computed seamlessly or split cleanly without overriding final completion.

**Proven Invariants:**
✓ Flow Extension never creates a second Logical Focus Session.
✓ Reset during Flow never revokes completed Work.
✓ Split fragments cannot revoke completed Work.
✓ Statistics operate on Logical Focus Sessions rather than individual fragments.
